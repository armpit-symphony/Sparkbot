Sparkbot v1.6.39
=================

Product truth and MCP control-plane cleanup.

- Position Sparkbot and LIMA Robotics OS as one system:
  Sparkbot is the governed agentic assistant and command center;
  LIMA Robotics OS is the Local Intelligent Machine Agent runtime
  for robot skills and physical-world execution exposed through MCP.
- Add the Workstation Robo OS button and MCP control-plane panel.
- Add typed tool manifests for Sparkbot and LIMA tools with policy
  metadata: read-only, write, destructive, external-send, robot-motion,
  and secret-use.
- Show risk level, required secrets, dry-run posture, and live Sparkbot
  health signals from existing Guardian/skills APIs.
- Document no-hardware LIMA demos through replay/simulation commands.
- Add first-run server/CLI onboarding: `scripts/sparkbot-setup.sh`,
  `scripts/sparkbot-start.sh`, Docker Compose v1/v2 detection, guided
  provider/Ollama prompts, `.env.local` preservation, CLI setup fallback,
  and a clearer web Controls fallback when no usable model provider is
  configured.
- Fix SSH first-run setup input: provider key prompts are visible by default
  so paste works naturally, `--hide-input` is available for users who prefer
  hidden provider-key entry, `--from-env` imports exported provider keys, and
  direct provider key flags can seed setup without manual `.env.local` editing.
- Fix local Docker startup on legacy `docker-compose` 1.29.x by using
  v1-compatible `env_file` syntax and printing a legacy compatibility mode
  message when the launcher falls back from Compose v2.
- Add MiniMax to the server setup wizard provider prompts and direct setup
  flags.
- Add Docker buildx preflight checks and `--install-docker-plugins` so fresh
  Ubuntu installs get a clear plugin fix instead of a raw BuildKit/buildx
  failure.
- Make the chat-table enum migration idempotent for PostgreSQL so fresh and
  partially initialized installs do not fail on duplicate `usertype`.
- Improve prestart diagnostics with plain-English database env validation and
  failure context while avoiding provider secret logging.
- Make the local Docker frontend port configurable with
  `SPARKBOT_FRONTEND_PORT`, auto-select the next available port when 3000 is
  busy, and print the actual startup URL.
- Split Docker launcher startup into local and server modes:
  `--local` keeps the web UI bound to 127.0.0.1, while `--server` binds it to
  0.0.0.0, detects the public IP, prints the public URL, and warns operators to
  use firewall rules or reverse-proxy auth.
- Harden server mode authentication: server startup disables local auto-login,
  prompts for missing/default/placeholder/short passphrases before Compose
  starts, saves the accepted passphrase to `.env.local`, never prints it, and
  builds the frontend with local auto-login off so the login/passphrase screen
  gates workspace access.
- Make server passphrase onboarding SSH-safe: hidden input falls back to
  visible input when the terminal cannot accept hidden typing or paste,
  `--show-passphrase-input` makes troubleshooting explicit, `--passphrase`
  supports direct setup, and `--dry-run-setup` validates first-run prompts
  without starting Docker.
- Start Docker Compose detached by default and mirror only non-secret frontend
  bind/port values into root `.env` so Compose interpolation works even though
  provider secrets stay in `.env.local`.
- Let Docker `env_file` supply provider keys, generated `SECRET_KEY`, passphrase,
  and Guardian settings without Compose interpolation overriding them with
  empty strings or development defaults.
- Proxy same-origin `/api`, `/docs`, and `/redoc` through the frontend nginx
  container so server-mode browser sessions do not depend on localhost API
  access from the user's laptop.
- Refresh README, capabilities docs, release note, and public downloader
  version markers for 1.6.39.
